## 一、Spring Boot 3 核心概念

### 1.1 什么是 Spring Boot？

Spring Boot 是基于 Spring Framework 的**快速应用开发框架**，核心设计理念是"约定优于配置"（Convention over Configuration）。

**传统 Spring 项目 vs Spring Boot 项目：**

| 对比维度 | 传统 Spring | Spring Boot |
|---------|------------|------------|
| 项目搭建 | 手动配置 web.xml、applicationContext.xml | 一行 `SpringApplication.run()` |
| 依赖管理 | 手动添加 jar，处理版本冲突 | `spring-boot-starter-*` 一键引入 |
| 服务器部署 | 打 war 包，部署到外部 Tomcat | 内嵌 Tomcat，直接 `java -jar` |
| 配置管理 | 分散在多个 XML 文件 | 集中在 `application.yml` |

### 1.2 @SpringBootApplication 做了什么？

```java
@SpringBootApplication
public class CampusAnimalApplication {
    public static void main(String[] args) {
        SpringApplication.run(CampusAnimalApplication.class, args);
    }
}
```

`@SpringBootApplication` 是三个注解的组合：

1. **`@SpringBootConfiguration`** — 标记该类为配置类（可以定义 `@Bean`）
2. **`@EnableAutoConfiguration`** — 根据 classpath 中的 jar 包自动配置 Spring 组件
   - 例如：检测到 `mysql-connector-j` → 自动配置 DataSource
   - 检测到 `spring-boot-starter-web` → 自动配置 DispatcherServlet
3. **`@ComponentScan`** — 扫描当前包及子包中的 `@Component`、`@Service` 等注解

### 1.3 IoC 容器与依赖注入

**IoC（Inversion of Control，控制反转）：** 将对象的创建和管理权交给 Spring 容器，而不是在代码中 `new` 对象。

**DI（Dependency Injection，依赖注入）：** Spring 自动将依赖的对象注入到需要它的地方。

**三种注入方式：**

```java
// 1. 构造器注入（推荐 — 不可变性、易于测试）
@RequiredArgsConstructor  // Lombok：自动生成包含所有 final 字段的构造器
public class AnimalServiceImpl implements AnimalService {
    private final AnimalMapper animalMapper;  // Spring 自动注入
}

// 2. Setter 注入（可选依赖场景）
@Autowired
public void setAnimalMapper(AnimalMapper animalMapper) { ... }

// 3. 字段注入（不推荐 — 难以测试、隐藏依赖）
@Autowired
private AnimalMapper animalMapper;
```

**本项目统一使用构造器注入 + Lombok `@RequiredArgsConstructor`。**

### 1.4 Bean 的生命周期

```
1. 实例化（调用构造器）
2. 属性赋值（@Value、@Autowired）
3. 初始化（@PostConstruct、InitializingBean）
4. 就绪（可被使用）
5. 销毁（@PreDestroy、DisposableBean）
```

### 1.5 application.yml 配置原理

Spring Boot 启动时会自动加载 `application.yml`（或 `.properties`），并将配置值绑定到对应的对象：

```yaml
# 配置：
jwt:
  secret: Y2FtcHVz...

# Java 代码中读取：
@Value("${jwt.secret}")
private String secret;
```

Spring Boot 的自动配置类（如 `DataSourceAutoConfiguration`）也是从这里读取配置。例如 `spring.datasource.url` 这个配置项，就是被 `DataSourceAutoConfiguration` 读取并创建 DataSource Bean。

## 二、MyBatis-Plus 持久层框架

### 2.1 MyBatis vs MyBatis-Plus

**MyBatis** 是半自动化的 ORM 框架，需要手写 SQL 映射（XML 文件或注解）。

**MyBatis-Plus** 在 MyBatis 基础上做了增强，提供了：
- **BaseMapper**：内置 17 个 CRUD 方法，无需写一行 SQL
- **LambdaQueryWrapper**：用 Lambda 表达式构建查询条件，避免字段名字符串硬编码
- **分页插件**：自动改写 SQL 添加 LIMIT

### 2.2 BaseMapper 提供了什么？

```java
public interface UserMapper extends BaseMapper<User> {
    // 无需写任何方法！自动拥有：
}

// 使用示例：
userMapper.selectById(1L);                    // 按主键查
userMapper.selectList(wrapper);               // 条件查列表
userMapper.selectPage(page, wrapper);         // 分页查
userMapper.insert(user);                      // 插入
userMapper.updateById(user);                  // 按主键更新
userMapper.deleteById(1L);                    // 按主键删除
userMapper.selectCount(wrapper);              // 计数
```

### 2.3 LambdaQueryWrapper 为什么好？

**错误做法（字符串硬编码）：**
```java
// 字段名写死为字符串 → 重命名字段时不会自动更新 → BUG
QueryWrapper<User> wrapper = new QueryWrapper<>();
wrapper.eq("username", "admin");
```

**正确做法（Lambda 表达式）：**
```java
// Lambda 引用 → IDE 可以自动重构
LambdaQueryWrapper<User> wrapper = new LambdaQueryWrapper<>();
wrapper.eq(User::getUsername, "admin");
```

### 2.4 分页插件原理

```
Page<Animal> page = new Page<>(1, 10);  // 第1页，每页10条
Page<Animal> result = animalMapper.selectPage(page, wrapper);

result.getTotal();     // 总记录数
result.getRecords();   // 当前页数据
result.getCurrent();   // 当前页码
result.getPages();     // 总页数
```

**背后发生了什么：**

1. 拦截器 `PaginationInnerInterceptor` 拦截 SQL 执行
2. 先执行 COUNT 查询获取总数：`SELECT COUNT(*) FROM animal WHERE ...`
3. 再执行分页查询：`SELECT * FROM animal WHERE ... LIMIT 0, 10`
4. 将结果封装到 Page 对象中

这就是为什么必须在 `MybatisPlusConfig` 中注册拦截器。

### 2.5 驼峰命名映射

```yaml
mybatis-plus:
  configuration:
    map-underscore-to-camel-case: true
```

此配置自动将数据库下划线字段映射到 Java 驼峰属性：

| 数据库字段 | Java 属性 |
|-----------|----------|
| `created_at` | `createdAt` |
| `real_name` | `realName` |
| `cover_image` | `coverImage` |

## 三、Spring Security + JWT 认证体系

### 3.1 为什么用 JWT 而不是 Session？

| 对比维度 | Session（有状态） | JWT（无状态） |
|---------|-----------------|-------------|
| 存储位置 | 服务端内存/Redis | 客户端（localStorage） |
| 扩展性 | 需共享 Session | 天然支持水平扩展 |
| 跨域 | 需要额外配置 | 天然支持（请求头传递） |
| 注销 | 服务端删除 Session | 客户端删除 token |

### 3.2 JWT 结构

JWT 由三段 Base64 编码的 JSON 组成，用 `.` 分隔：

```
eyJhbGciOi... . eyJzdWIiOi... . SflKxwRJS...
   Header        Payload       Signature
```

**Header（头部）：** 签名算法（HS256）+ 令牌类型（JWT）
```json
{ "alg": "HS256", "typ": "JWT" }
```

**Payload（荷载）：** 业务数据（Claims）
```json
{
  "sub": "admin",           // subject = 用户名
  "userId": 1,              // 自定义声明
  "role": "ADMIN",          // 自定义声明
  "iat": 1717800000,        // 签发时间
  "exp": 1717886400         // 过期时间（24小时后）
}
```

**Signature（签名）：** 用密钥对前两段进行 HMAC-SHA256 计算，防止被篡改。

### 3.3 认证流程（完整）

```
┌──────────┐     ┌──────────────┐     ┌──────────────┐
│  前端    │     │  SecurityConfig │   │  JwtFilter   │
└────┬─────┘     └──────┬───────┘     └──────┬───────┘
     │                  │                    │
     │ 1. POST /api/auth/login              │
     │─────────────────►│                    │
     │                  │ 放行（白名单）       │
     │                  │───────────────────►│
     │                  │                    │ 无 token，放行
     │                  │◄───────────────────│
     │  返回 JWT token   │                    │
     │◄─────────────────│                    │
     │                  │                    │
     │ 2. GET /api/animals (带 Authorization 头)
     │─────────────────►│                    │
     │                  │ 需要认证            │
     │                  │───────────────────►│
     │                  │                    │ 提取 Bearer token
     │                  │                    │ 解析 JWT
     │                  │                    │ 存入 SecurityContext
     │                  │◄───────────────────│
     │                  │ 验证权限            │
     │  返回数据         │                    │
     │◄─────────────────│                    │
```

### 3.4 Spring Security 过滤器链

```
请求进入
  │
  ├── SecurityContextPersistenceFilter    (管理 SecurityContext)
  ├── JwtAuthenticationFilter  ◄── 我们的过滤器，在这里解析 JWT
  ├── UsernamePasswordAuthenticationFilter  (表单登录，本项目未使用)
  ├── FilterSecurityInterceptor            (授权判断 — 检查权限)
  ▼
Controller 处理请求
```

**关键设计：JWT 过滤器不拦截请求。** 即使 token 无效，过滤器也只是不设置认证信息，然后放行。真正的拦截由 `SecurityConfig` 中的 `.anyRequest().authenticated()` 完成 — 如果此时 SecurityContext 中没有认证信息，Spring Security 会返回 401。

### 3.5 SecurityContextHolder

Spring Security 使用 **ThreadLocal** 存储安全上下文，这意味着：

```java
// 同一线程的任何地方都能获取当前用户
TokenUser user = SecurityUtils.getCurrentUser();
```

**ThreadLocal 原理：** 每个线程有自己的独立副本，不同线程互相隔离。所以同一请求的 Controller → Service → Mapper 调用链中，都能拿到同一个用户信息。

## 四、Vue 3 前端框架

### 4.1 Composition API vs Options API

**Options API（Vue 2 风格）：**
```javascript
export default {
  data() { return { count: 0 } },
  methods: { increment() { this.count++ } },
  mounted() { console.log('mounted') }
}
```

**Composition API（本项目使用）：**
```javascript
// <script setup> — 编译时语法糖，无需手动 return
const count = ref(0)
function increment() { count.value++ }
onMounted(() => console.log('mounted'))
```

Composition API 的优势：
- 逻辑可以按功能组织（而非按选项类型）
- 更好的 TypeScript 支持
- 易于提取和复用逻辑（组合函数）

### 4.2 响应式系统：ref 与 reactive

```javascript
// ref — 包装基本类型，用 .value 访问
const count = ref(0)
count.value++  // 模板中自动解包，只需写 {{ count }}

// reactive — 包装对象，直接访问属性
const form = reactive({ name: '', type: '' })
form.name = '小花'  // 直接访问，无需 .value
```

**选择建议：**
- 基本类型 → `ref`
- 对象/数组 → `reactive`（访问更自然）
- 需要替换整个对象 → `ref`

### 4.3 组件通信

**父 → 子（Props）：**
```javascript
// 子组件定义
const props = defineProps({
  animal: { type: Object, required: true }
})
// 父组件传递
<AnimalCard :animal="animalObject" />
```

**状态共享（Pinia Store）：**
```javascript
// 任何组件都能访问
const auth = useAuthStore()
console.log(auth.isLoggedIn)
```

### 4.4 Pinia 状态管理

Pinia 是 Vue 3 官方状态管理库：

```javascript
export const useAuthStore = defineStore('auth', () => {
  // state = ref / reactive
  const token = ref(localStorage.getItem('token') || '')

  // getter = computed
  const isLoggedIn = computed(() => !!token.value)

  // action = 普通函数
  async function login(username, password) { ... }

  return { token, isLoggedIn, login }
})
```

### 4.5 Vue Router 导航守卫

```javascript
router.beforeEach((to, from, next) => {
  // to：目标路由
  // from：来源路由
  // next()：放行
  // next('/login')：重定向

  if (to.meta.auth && !auth.token) return next('/login')
  if (to.meta.admin && auth.user?.role !== 'ADMIN') return next('/')
  next()
})
```

### 4.6 Axios 拦截器

```
请求发出前 → 请求拦截器（加 token） → 服务端 → 响应拦截器（处理错误） → 组件接收
```

## 五、项目分层架构

```
┌─────────────────────────────────────────┐
│              Controller 层               │
│  - 接收 HTTP 请求                        │
│  - 参数校验（@Valid）                     │
│  - 调用 Service                          │
│  - 返回 Result<T>                        │
├─────────────────────────────────────────┤
│               Service 层                 │
│  - 业务逻辑                              │
│  - 事务管理                              │
│  - DTO → Entity 转换                    │
│  - Entity → VO 转换                     │
├─────────────────────────────────────────┤
│               Mapper 层                  │
│  - 数据库 CRUD                           │
│  - BaseMapper 内置方法                   │
│  - 自定义 SQL（如需要）                   │
├─────────────────────────────────────────┤
│               Entity 层                  │
│  - 与数据库表一一映射                      │
│  - @TableName, @TableId                 │
└─────────────────────────────────────────┘

数据流转方向：
  Request → DTO → Service → Entity → Mapper → Database
  Database → Mapper → Entity → Service → VO → Response
```

**单向依赖原则：** Controller → Service → Mapper，不可反向调用。

### DTO vs VO vs Entity

| 类型 | 全称 | 用途 | 方向 |
|------|------|------|------|
| **DTO** | Data Transfer Object | 接收前端请求参数 | 前端 → 后端 |
| **Entity** | 实体 | 映射数据库表 | 后端 ↔ 数据库 |
| **VO** | View Object | 返回给前端的数据 | 后端 → 前端 |

**为什么不直接用 Entity 接收请求和返回响应？**

1. 安全：Entity 可能包含敏感字段（如密码），不能暴露给前端
2. 灵活：DTO/VO 可以跨表组合多个实体的字段
3. 校验：DTO 可以添加 `@NotBlank` 等校验注解
4. 解耦：前端字段变更不影响数据库表结构

## 六、一次请求的完整生命周期

以"查看动物详情"（`GET /api/animals/1`）为例，追踪一次完整请求：

```
1. 浏览器发起请求
   GET http://localhost:5173/api/animals/1
   Authorization: Bearer eyJhbGci...

2. Vite 开发服务器（端口 5173）
   - 匹配到 /api 前缀 → 代理到 localhost:8080

3. Tomcat 接收请求（端口 8080）
   - 请求进入 Spring 的 DispatcherServlet

4. Spring Security 过滤器链
   ├── JwtAuthenticationFilter
   │   - extractToken() → "eyJhbGci..."
   │   - jwtTokenProvider.parseToken() → TokenUser(userId=1, username="admin", role="ADMIN")
   │   - 创建 UsernamePasswordAuthenticationToken → 存入 SecurityContextHolder
   ├── FilterSecurityInterceptor
   │   - 检查权限：GET /api/animals/{id} 只需认证，不需要 ADMIN
   │   - 通过 ✓

5. DispatcherServlet 路由
   - 匹配到 AnimalController.getDetail(1)
   - 从 URL 路径中提取 id=1

6. Controller 层
   AnimalController.getDetail(1)
   → return Result.success(animalService.getDetail(1))

7. Service 层
   AnimalServiceImpl.getDetail(1)
   ├── animalMapper.selectById(1)           → Animal{id=1, name="小花", ...}
   ├── checkInMapper.selectList(wrapper)    → List<CheckIn> (3条打卡记录)
   ├── 收集 userIds → {2, 3, 5}
   ├── userMapper.selectBatchIds({2,3,5})   → Map<id, username>
   └── 组装 AnimalDetailVO（复制 Animal 属性 + 填充 timeline）

8. 返回结果
   Result<AnimalDetailVO>(200, "操作成功", animalDetailVO)

9. Jackson 序列化
   - AnimalDetailVO → JSON 字符串

10. HTTP 响应返回
    HTTP 200 OK
    Content-Type: application/json
    {
      "code": 200,
      "message": "操作成功",
      "data": {
        "id": 1,
        "name": "小花",
        "type": "猫",
        ...
        "timeline": [
          { "id": 10, "username": "张三", "content": "在图书馆看到它", ... },
          ...
        ]
      }
    }

11. Axios 响应拦截器
    - 检查 code === 200 → 成功
    - 返回 response.data 给组件

12. Vue 组件
    - animal.value = response.data
    - 模板重新渲染 → 用户看到页面
```

## 附录：常用注解速查表

### Spring Boot

| 注解 | 位置 | 作用 |
|------|------|------|
| `@SpringBootApplication` | 启动类 | 三合一：配置 + 自动配置 + 组件扫描 |
| `@RestController` | Controller | 组合 @Controller + @ResponseBody（自动 JSON） |
| `@RequestMapping("/api/xxx")` | Controller | 路径前缀 |
| `@GetMapping / @PostMapping / @PutMapping / @DeleteMapping` | 方法 | HTTP 方法映射 |
| `@PathVariable` | 参数 | 从 URL 路径取参数 |
| `@RequestParam` | 参数 | 从 Query String / Form Data 取参数 |
| `@RequestBody` | 参数 | 从请求体取 JSON |
| `@Valid` | 参数 | 触发 Jakarta Bean Validation |
| `@Service` | Service 实现 | 标记为 Service 层 Bean |
| `@Component` | 工具类 | 标记为通用 Bean |
| `@Configuration` | 配置类 | 标记为配置类（可含 @Bean） |
| `@Bean` | 方法 | 方法返回值注册到 IoC 容器 |
| `@Value("${xxx}")` | 字段 | 读取 application.yml 配置 |
| `@RequiredArgsConstructor` | 类 | Lombok：生成 final 字段的构造器（构造器注入） |

### Spring Security

| 注解 | 位置 | 作用 |
|------|------|------|
| `@EnableWebSecurity` | 配置类 | 启用 Web 安全 |
| `@EnableMethodSecurity` | 配置类 | 启用方法级权限（@PreAuthorize） |
| `@PreAuthorize("hasRole('ADMIN')")` | 方法 | 方法执行前校验角色 |

### MyBatis-Plus

| 注解 | 位置 | 作用 |
|------|------|------|
| `@TableName("xxx")` | Entity | 指定映射的表名 |
| `@TableId(type = IdType.AUTO)` | Entity.id | 主键自增 |
| `@Mapper` | Mapper 接口 | MyBatis 扫描识别 |

### Jakarta Validation

| 注解 | 作用 |
|------|------|
| `@NotBlank` | 字符串非空且不能全是空白 |
| `@NotNull` | 值不能为 null |
| `@Size(min, max)` | 字符串长度限制 |
| `@Min(value)` | 数值最小值 |
