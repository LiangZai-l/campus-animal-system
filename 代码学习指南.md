## 二、通用模块（common）— 基础设施

### 2.1 ResultCode — 状态码常量接口

```java
public interface ResultCode {
    int SUCCESS = 200;
    int BAD_REQUEST = 400;
    int UNAUTHORIZED = 401;
    int FORBIDDEN = 403;
    int NOT_FOUND = 404;
    int CONFLICT = 409;
    int INTERNAL_ERROR = 500;
}
```

**设计要点：**
- 使用 `interface` 而非 `class`：接口字段默认为 `public static final`，天然适合常量定义
- 覆盖了 RESTful API 最常用的状态码：成功的 200，客户端的 4xx，服务端的 5xx
- 409 (CONFLICT)：用于"用户名已存在"这类资源冲突场景，比通用的 400 更具语义

### 2.2 Result\<T\> — 统一响应体

```java
public class Result<T> {
    private int code;
    private String message;
    private T data;
}
```

**核心设计：**

1. **泛型 `<T>`**：让不同类型的业务数据都能用同一个响应体。例如 `Result<Void>` 表示无数据，`Result<LoginVO>` 表示登录响应。

2. **静态工厂方法**：不直接用 `new Result(...)`，而是通过命名方法创建：
   - `Result.success(data)` → 成功 + 数据
   - `Result.success()` → 成功无数据（内部调用 `success(null)`）
   - `Result.error(code, message)` → 指定错误码
   - `Result.error(message)` → 默认 500 错误

3. **前端判断逻辑**：前端统一检查 `code === 200` 判断成功与否，不需要关心 HTTP 状态码。

### 2.3 BusinessException — 自定义业务异常

```java
public class BusinessException extends RuntimeException {
    private final int code;
}
```

**为什么要有 BusinessException？**

- 区别于系统异常（NullPointerException、SQLException）—— 业务异常是"预期中的错误"
- 在 `GlobalExceptionHandler` 中对业务异常只打 WARN 日志（不打印堆栈），系统异常打 ERROR 日志（打印完整堆栈）

### 2.4 GlobalExceptionHandler — 全局异常处理

```java
@RestControllerAdvice  // = @ControllerAdvice + @ResponseBody
public class GlobalExceptionHandler {
    @ExceptionHandler(BusinessException.class)
    public Result<Void> handleBusinessException(BusinessException e) { ... }

    @ExceptionHandler(Exception.class)  // 兜底
    public Result<Void> handleException(Exception e) { ... }
}
```

**工作原理：**
1. `@RestControllerAdvice` 是对所有 Controller 的 AOP 增强
2. 当任何 Controller 抛出异常时，Spring 会查找匹配的 `@ExceptionHandler`
3. 方法返回值自动序列化为 JSON（因为 `@RestControllerAdvice`）

**异常匹配优先级：** Spring 匹配最具体的类型。例如抛出 `BusinessException`，不会匹配到 `Exception` 处理器（尽管 BusinessException 也是 Exception）。

## 三、安全模块（security）— 认证授权

### 3.1 TokenUser — 用户主体

```java
public class TokenUser {
    private final Long userId;
    private final String username;
    private final String role;
}
```

这是一个轻量级的用户主体对象，只存储从 JWT 中解析出的必需字段。与 Spring Security 的 `UserDetails` 不同，它不存储密码，因为 JWT 认证天然不需要在内存中保存密码。

### 3.2 JwtTokenProvider — JWT 核心

**构造器：** 从 `application.yml` 读取配置 →
```java
public JwtTokenProvider(
    @Value("${jwt.secret}") String secret,       // Base64 密钥
    @Value("${jwt.expiration}") long expiration   // 过期时间（ms）
) {
    this.key = Keys.hmacShaKeyFor(Decoders.BASE64.decode(secret));
}
```

Base64 解码后创建 HMAC-SHA256 密钥。为什么用 Base64？因为密钥是二进制数据，Base64 编码后可以安全放在 YAML 文件中。

**generateToken：** 构建 JWT 的三段式结构。
```java
Jwts.builder()
    .subject(username)           // sub — 标准声明
    .claim("userId", userId)     // 自定义声明
    .claim("role", role)         // 自定义声明
    .issuedAt(now)               // iat — 签发时间
    .expiration(expiry)          // exp — 过期时间
    .signWith(key)               // 签名
    .compact();                  // 生成字符串
```

**parseToken：** 解析并验证 JWT。
```java
Jwts.parser()
    .verifyWith(key)             // 验签 + 验过期
    .build()
    .parseSignedClaims(token)
    .getPayload();               // 提取 Claims
```

如果 token 过期、签名不对、或被篡改，`jjwt` 库会自动抛出异常，由调用方（`JwtAuthenticationFilter`）catch 处理。

### 3.3 JwtAuthenticationFilter — 请求拦截

```java
public class JwtAuthenticationFilter extends OncePerRequestFilter { ... }
```

**核心流程：**
```
1. extractToken() — 从 Authorization: Bearer xxx 中提取 token
2. 如果 token 存在 → jwtTokenProvider.parseToken()
3. 解析成功 → 创建 Authentication → 存入 SecurityContextHolder
4. 解析失败 → 不设认证，放行（后续 SecurityConfig 返回 401）
5. filterChain.doFilter() — 继续执行其他过滤器
```

**为什么继承 OncePerRequestFilter？** 相比于 `GenericFilterBean`，它能保证同一请求内只执行一次（处理 forward/include 的重复执行问题）。

### 3.4 SecurityUtils — 安全工具

```java
public class SecurityUtils {
    public static TokenUser getCurrentUser() {
        Authentication auth = SecurityContextHolder.getContext().getAuthentication();
        if (auth != null && auth.getPrincipal() instanceof TokenUser tokenUser) {
            return tokenUser;
        }
        return null;
    }
}
```

**关键点：`SecurityContextHolder` 使用 ThreadLocal 存储上下文。**
这意味着同一个 HTTP 请求线程内的 Controller → Service → Mapper 任何地方调用 `SecurityUtils.getCurrentUser()`，都能拿到同一个用户信息。

## 四、配置模块（config）— 系统设定

### 4.1 SecurityConfig — Spring Security 配置

这是整个系统安全策略的总控制器。逐段解读：

```java
// 1. 禁用 CSRF（前后端分离不需要）
.csrf(csrf -> csrf.disable())

// 2. 无状态会话（不创建 HttpSession，每次请求都通过 JWT 认证）
.sessionManagement(session ->
    session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))

// 3. 认证失败返回 JSON（而非默认的 302 重定向到登录页）
.exceptionHandling(ex -> ex
    .authenticationEntryPoint(...)   // 401 时返回 JSON
    .accessDeniedHandler(...)        // 403 时返回 JSON
)

// 4. URL 访问规则
.authorizeHttpRequests(auth -> auth
    .requestMatchers(HttpMethod.GET, "/api/files/**").permitAll()  // 上传图片公开
    .requestMatchers("/api/auth/register", "/api/auth/login").permitAll()  // 注册登录公开
    .requestMatchers("/doc.html", "/swagger-ui/**", ...).permitAll()  // API 文档公开
    .requestMatchers("/error").permitAll()  // 错误页面公开
    .anyRequest().authenticated()  // 其余全部需要认证
)

// 5. 插入 JWT 过滤器
.addFilterBefore(jwtAuthenticationFilter, UsernamePasswordAuthenticationFilter.class)
```

### 4.2 DataInitializer — 管理员自动创建

```java
public class DataInitializer implements CommandLineRunner {
    public void run(String... args) {
        User admin = userMapper.selectOne(
            new LambdaQueryWrapper<User>().eq(User::getUsername, "admin"));
        if (admin == null) {
            // 创建管理员...
        }

}
```

**为什么用 `CommandLineRunner` 而非 SQL 脚本？**
- 密码需要 BCrypt 加密，SQL 脚本无法生成
- 幂等性：每次重启检查 admin 是否存在，已存在则跳过

## 六、数据传输（dto + vo）— 出入参规范

### DTO：接收前端数据

DTO 带有 `@NotBlank`、`@NotNull`、`@Size`、`@Min` 等校验注解，在 Controller 中通过 `@Valid` 触发校验。

```java
@Data
public class RegisterDTO {
    @NotBlank(message = "用户名不能为空")
    @Size(min = 3, max = 50, message = "用户名长度需在3-50之间")
    private String username;
    ...
}
```

**校验失败时：** Spring 抛出 `MethodArgumentNotValidException` → `GlobalExceptionHandler` 捕获 → 提取各字段的错误信息 → 拼接后返回 400。

### VO：返回给前端的数据

VO 不包含敏感字段，可以跨表聚合多个实体的数据。

**AnimalDetailVO 的继承设计：**
```java
@Data
@EqualsAndHashCode(callSuper = true)
public class AnimalDetailVO extends AnimalVO {
    private List<CheckInVO> timeline;  // 额外增加打卡时间轴
}
```
列表查询返回 `AnimalVO`（不加载 timeline），详情查询返回 `AnimalDetailVO`（含 timeline）。这样列表查询的数据量更小。

**CheckInVO 的 username 字段：** username 不在 `check_in` 表中，是关联 `sys_user` 表查询获得的。Service 层通过批量查询避免 N+1 问题。

## 七、业务逻辑（service）— 核心业务

### 7.1 AuthServiceImpl — 认证逻辑

**注册流程（register）：**
```
1. 查重（selectCount by username） → 已存在则抛 CONFLICT
2. BCrypt 加密密码 → 插入用户 → 角色固定为 "USER"
```

**为什么注册的角色固定为 USER？** 管理员由 DataInitializer 在启动时创建，普通注册永远只能获得 USER 权限。这样没有人能通过注册页面创建 ADMIN 账号。

**登录流程（login）：**
```
1. 查用户（selectOne by username）
2. 用户不存在 → 抛 "用户名或密码错误"（不区分不存在和密码错误）
3. 状态 != 1 → 抛 "账号已被禁用"
4. BCrypt.matches(明文, 密文) → 不匹配 → 抛 "用户名或密码错误"
5. jwtTokenProvider.generateToken() → 生成 JWT
6. 构建 LoginVO 返回（含 token + userId + username + realName + role）
```

**安全设计：登录失败的提示统一为"用户名或密码错误"，不区分具体原因。** 这样攻击者无法通过错误信息判断哪些用户名是有效的（防止用户名枚举攻击）。

### 7.2 AnimalServiceImpl — 动物档案 CRUD

**新增（create）：**
```java
animal.setCreatedBy(SecurityUtils.getCurrentUserId());
```
`createdBy` 从 `SecurityContext` 获取当前登录用户 ID，不从前端传入。这样即使前端伪造了 `createdBy`，后端也以 JWT 中的用户为准。

**分页查询（page）：**
```java
LambdaQueryWrapper<Animal> wrapper = new LambdaQueryWrapper<>();
if (StringUtils.hasText(query.getName())) {
    wrapper.like(Animal::getName, query.getName());  // 模糊搜索
}
if (StringUtils.hasText(query.getType())) {
    wrapper.eq(Animal::getType, query.getType());    // 精确筛选
}
wrapper.orderByDesc(Animal::getCreatedAt);
```

动态构建查询条件：只有前端传了 `name` 参数才加 LIKE 条件，只有传了 `type` 参数才加 EQ 条件。

**详情查询（getDetail）— 核心优化：避免 N+1 查询：**
```java
// ❌ 错误做法 — 100 条打卡需要 101 次数据库查询
for (CheckIn checkIn : checkIns) {
    User u = userMapper.selectById(checkIn.getUserId());  // N 次查询！
    vo.setUsername(u.getUsername());
}

// ✅ 正确做法 — 总共 3 次查询
Set<Long> userIds = checkIns.stream()
    .map(CheckIn::getUserId)
    .collect(Collectors.toSet());  // 收集 userId，去重

Map<Long, String> usernameMap = userMapper.selectBatchIds(userIds)  // 1 次批量查询
    .stream()
    .collect(Collectors.toMap(User::getId, User::getUsername));

// 组装 VO 时从 Map 中取 username（O(1)）
vo.setUsername(usernameMap.get(ci.getUserId()));
```

### 7.3 FileServiceImpl — 文件上传

**安全设计：**
1. **文件类型校验**：`contentType.startsWith("image/")` — 只允许图片
2. **扩展名推导**：用 JDK 21 的 `switch` 表达式从 Content-Type 推导扩展名，不信任用户上传的文件名
3. **UUID 文件名**：防冲突、防路径遍历、防预测

```java
// JDK 21 switch 表达式
String extension = switch (contentType) {
    case "image/jpeg" -> ".jpg";
    case "image/png"  -> ".png";
    case "image/gif"  -> ".gif";
    case "image/webp" -> ".webp";
    default           -> ".jpg";
};
```

### 7.4 CheckInServiceImpl — 打卡业务

**关键安全设计：userId 从 SecurityContext 获取。**
```java
checkIn.setUserId(SecurityUtils.getCurrentUserId());
```
打卡人不由前端指定，而是从 JWT 令牌中提取，防止用户替别人打卡。

**toVOList 方法：** 同样是批量查询避免 N+1，与 `AnimalServiceImpl.getDetail()` 的策略一致。

## 八、控制器（controller）— API 入口

### Controller 的唯一职责

Controller 只做三件事，**不包含任何业务逻辑**：

1. 接收参数（`@RequestBody`、`@PathVariable`、`@RequestParam`）
2. 调用 Service
3. 包装为 `Result<T>` 返回

```java
@PostMapping("/register")
public Result<Void> register(@Valid @RequestBody RegisterDTO registerDTO) {
    authService.register(registerDTO);  // 委托给 Service
    return Result.success();            // 统一包装
}
```

### @PreAuthorize 方法级权限

```java
@PostMapping
@PreAuthorize("hasRole('ADMIN')")  // 只有 ADMIN 角色才能访问
public Result<AnimalVO> create(@Valid @RequestBody AnimalSaveDTO dto) { ... }
```

`@EnableMethodSecurity`（在 SecurityConfig 上）启用了这个注解。当非 ADMIN 用户调用此接口时，Spring Security 抛出 `AccessDeniedException` → `GlobalExceptionHandler` 捕获 → 返回 403。

### RESTful URL 设计

| HTTP 方法 | URL | 操作 |
|----------|-----|------|
| GET | /api/animals | 列表查询 |
| GET | /api/animals/{id} | 查看详情 |
| POST | /api/animals | 新增 |
| PUT | /api/animals/{id} | 修改 |
| DELETE | /api/animals/{id} | 删除 |

URL 使用**名词复数** + HTTP 方法表达语义，不出现动词（如 `/api/animals/create` 是不规范的）。

**响应拦截器：** 双回调模式。
```javascript
api.interceptors.response.use(

        if (data.code === 200) return data  // 解包
        ElMessage.error(data.message)       // 业务错误弹窗
        return Promise.reject(...)
    },
    error => {  // 非 2xx

            // 清除 token，跳转登录页
            window.location.href = '/login'
        }
        ...
    }
)
```

### 9.3 路由守卫（router/index.js）

```javascript
router.beforeEach((to, from, next) => {
    const auth = useAuthStore()

    if (to.meta.auth && !auth.token) return next('/login')    // 需登录
    if (to.meta.admin && auth.user?.role !== 'ADMIN') return next('/')  // 需管理员
    if (to.meta.guest && auth.token) return next('/')         // 已登录访客
    next()
})
```

利用 `meta` 字段标记路由的权限要求，守卫中统一判断。

### 9.4 Pinia 认证 Store（stores/auth.js）

```javascript
export const useAuthStore = defineStore('auth', () => {
    const token = ref(localStorage.getItem('token') || '')
    const user = ref(safeParseUser())

    // 计算属性
    const isLoggedIn = computed(() => !!token.value)
    const isAdmin = computed(() => user.value?.role === 'ADMIN')

    // 登录
    async function login(username, password) {
        const res = await api.post('/auth/login', { username, password })
        token.value = res.data.token
        user.value = { ... }
        localStorage.setItem('token', token.value)   // 持久化
        localStorage.setItem('user', JSON.stringify(user.value))
    }

    // 注销
    function logout() {
        token.value = ''
        user.value = null
        localStorage.removeItem('token')
        localStorage.removeItem('user')
    }

    return { token, user, isLoggedIn, isAdmin, login, register, logout }
})
```

**持久化策略：** token 和 user 存储在 localStorage 中，页面刷新后自动恢复登录状态。

## 十一、开发中的关键设计决策

### 决策 1：为什么 DTO/VO/Entity 三层分离？

**不分离的问题：**
- 前端可以直接看到 Entity 中不想暴露的字段（如密码）
- 前端字段命名变更会导致数据库表结构变更
- 跨表数据（如打卡记录中的用户名）无法在 Entity 中表达

**分离后：**
- Entity 只关心数据库映射
- DTO 只包含前端传入的字段 + 校验规则
- VO 可以跨表聚合，字段自由组合

### 决策 2：为什么批量查询避免 N+1？

打卡列表场景：100 条打卡记录，每条需要发布者的用户名。

| 方案 | SQL 次数 | 时间复杂度 |
|------|---------|-----------|
| N+1（循环查询） | 101 次 | O(N) 次数据库往返 |
| 批量查询 | 3 次 | 固定 3 次 |

数据库网络往返是昂贵的操作，批量查询将 N 次往返合并为 1 次。

### 决策 3：为什么 userId 从 SecurityContext 取而非从前端传？

```java
// ❌ 不安全：前端可以伪造 userId
checkIn.setUserId(dto.getUserId());

// ✅ 安全：userId 来自 JWT，前端无法伪造
checkIn.setUserId(SecurityUtils.getCurrentUserId());
```

如果从前端传 userId，任何人都可以通过修改请求数据来冒充他人打卡。从 JWT 中提取的 userId 是经过签名验证的，无法伪造。

### 决策 4：为什么 JWT 存 localStorage 而不是 Cookie？

| 存储方式 | 优点 | 缺点 |
|---------|------|------|
| localStorage | 不自动发送，防 CSRF | XSS 攻击可读取 |
| httpOnly Cookie | XSS 不可读取 | 需后端配合，CSRF 需额外防护 |

学习项目中选用 localStorage，因为：
- 实现简单，前端完全控制
- 不存在 CSRF 风险（手动添加 Authorization 头）
- XSS 防护可通过 CSP（Content Security Policy）缓解

### 决策 5：为什么文件扩展名从 Content-Type 推导？

```java
// ❌ 危险：使用用户上传的原始文件名
String filename = file.getOriginalFilename();  // 可能是 "../../../etc/passwd.jpg"

// ✅ 安全：从 Content-Type 推导扩展名
String extension = switch (contentType) { ... };
String filename = UUID.randomUUID() + extension;  // a1b2c3d4.jpg
```

用户上传的文件名可能包含路径遍历攻击字符串，直接使用可能导致文件写入任意目录。

### 决策 6：为什么登录失败不区分用户不存在和密码错误？

攻击者可以通过错误信息差异来枚举系统中的有效用户名：
- 如果"用户不存在"和"密码错误"提示不同 → 攻击者可以逐个尝试用户名，确认哪些用户存在
- 统一提示"用户名或密码错误" → 攻击者无法判断是哪种情况
